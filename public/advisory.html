منهجية التقييم المتدرجة

1

الاستطلاع السلبي (Passive Reconnaissance)

البحث في مصادر مفتوحة — قنوات Telegram العامة، المنتديات، ومحركات البحث — عن بيانات اعتماد MIS المسربة دون أي تفاعل مباشر مع الأنظمة.

2

تحليل بيانات الاعتماد

تصنيف ما تم جمعه، تحديد الحسابات النشطة، واستخلاص نمط التسمية المستخدم.

3

تحليل آلية إعادة تعيين كلمة المرور

فحص منطق Reset Password وتحديد الثغرة المنطقية في نظام أسئلة الأمان (Knowledge-Based Authentication).

4

إثبات المفهوم المحكوم (Controlled PoC)

تطبيق عملي على حساب واحد بإذن وعلم المشرف لإثبات قابلية الاستغلال الفعلي.

5

الإفصاح المسؤول الفوري (Responsible Disclosure)

إخطار المشرف الأكاديمي في نفس يوم الاكتشاف، ثم إعداد هذا التقرير الشامل للإدارة.

VULN-002 آلية إعادة تعيين كلمة المرور — أسئلة أمان قابلة للتخمين CRITICAL CVSS 9.1 CWE-640 ▼

آلية Password Reset تعتمد على Knowledge-Based Authentication (KBA) — وهي أسئلة شخصية مثل الاسم الكامل أو تاريخ الميلاد. المشكلة الجوهرية أن هذه المعلومات متاحة علنًا في دليل موظفي الجامعة أو وسائل التواصل الاجتماعي، مما يجعل هذا النظام معادلًا أمنيًا لعدم وجود أي حماية على الإطلاق.

أسئلة مكتشفة في النظام

Q1 — "ما اسمك الكامل؟" → متاح في دليل موظفي الجامعة
Q2 — "ما تاريخ ميلادك؟" → متاح على Facebook / LinkedIn
Q3 — "ما اسم مسقط رأسك؟" → معلومة عامة في كثير من الأحيان
Q4 — "ما اسم والدتك قبل الزواج؟" → قابل للحصول من سجلات قسم الموارد البشرية
🚫

موقف OWASP الرسمي: ترفض OWASP صراحةً KBA كآلية استرداد كلمة مرور. يُعدّ هذا النوع من الأسئلة مهجورًا أمنيًا منذ 2012. هذا النظام يستخدمها كطريقة الوحيدة للاسترداد.

📊

NIST SP 800-63B: يُصنّف NIST هذه الأسئلة ضمن أضعف مستويات التحقق ويوصي بالتخلص منها نهائيًا في أي نظام يحوي بيانات حساسة.

VULN-003 غياب كامل لـ Rate Limiting و Account Lockout على Password Reset HIGH CVSS 8.0 CWE-307 ▼

لا يوجد أي حد على عدد محاولات Password Reset. لا Rate Limiting، لا CAPTCHA، ولا قفل للحساب بعد محاولات فاشلة متعددة. هذا يُمكّن أي مهاجم من تنفيذ هجوم Brute-force آلي على أسئلة الأمان دون أي تأخير أو إنذار.

no_ratelimit_impact.txt 
# غياب Rate Limiting يعني:

محاولة 01: 1990-01-01 → خطأ
محاولة 02: 1991-03-15 → خطأ
محاولة 03: 1985-07-22 → خطأ
  ⚡ لا تأخير · لا قفل · لا إشعار
محاولة .N: [الإجابة الصحيحة] → ✓ نجاح!

# لتاريخ ميلاد: ~36500 تاريخ ممكن (100 عام)
# بدون Rate Limit: يمكن تجربتها في دقائق
# مع Rate Limit (5/ساعة): يلزم 300 يوم → غير عملي

⚡ يُمكّن الهجوم الآلي

سكريبت بسيط يجرب آلاف الاحتمالات في ثوانٍ

👻 لا رصد ولا إنذار

المستخدم الحقيقي لا يعلم بالمحاولات المتعددة

📉 يخفض العتبة الفنية

الهجوم ممكن لأي شخص بدون خبرة برمجية

VULN-004 غياب المصادقة متعددة العوامل (MFA) لجميع الأدوار HIGH CVSS 7.8 CWE-308 ▼

النظام يعتمد حصريًا على Single-Factor Authentication. لا يوجد عامل ثانٍ للتحقق لأي دور — حتى للمسؤولين وعمداء الكليات الذين لديهم وصول شامل لبيانات آلاف الطلاب. هذا يعني أن أي اختراق لكلمة المرور — سواء عبر تسريب أو هجوم Reset — يؤدي فورًا إلى سيطرة كاملة.

📊

إحصاء Microsoft Security 2024: تقلل MFA من خطر اختراق الحسابات بنسبة تتجاوز 99.9%. وجودها وحدها كان سيُبطل سيناريو الهجوم الكامل حتى لو نجحت جميع الثغرات الأخرى.

VULN-005 سياسة كلمات مرور ضعيفة — غياب متطلبات التعقيد MEDIUM CVSS 6.5 CWE-521 ▼

النظام لا يُطبّق متطلبات تعقيد إلزامية. بعد اختراق حساب عبر VULN-002، يمكن للمهاجم اختيار كلمة مرور بالغة البساطة مثل 123456 أو admin مما يجعل الحساب عرضة لمزيد من الاختراقات.

ما يُسمح به حاليًا ❌

  • كلمات مرور قصيرة (أقل من 8 أحرف)
  • كلمات بسيطة كـ 123456 و password
  • إعادة استخدام نفس كلمة المرور
  • لا انتهاء صلاحية إجبارية

المعيار المطلوب ✅

  • حد أدنى 12 حرفًا (NIST 2024)
  • مزيج من كبيرة وصغيرة وأرقام ورموز
  • فحص مقابل قواعد كلمات المرور المخترقة
  • تدوير إجباري كل 90 يومًا للمميزة
04 — سيناريو الهجوم

COMPLETE ATTACK CHAIN

الخطوات التالية توضح كيف تتسلسل الثغرات معًا لتُشكّل هجومًا كاملًا من الصفر إلى الاختراق التام، مع الإشارة إلى أي ثغرة يتم استغلالها في كل خطوة.

01

Reconnaissance — جمع أسماء المستخدمين

المهاجم يبحث في قنوات Telegram العامة وينجح في تجميع قائمة بأسماء مستخدمي MIS لكلية أو أكثر. يحدد الحسابات ذات الصلاحيات العالية عبر قراءة نمط التسمية.

→ VULN-001 exploited · المصدر: منصات عامة مفتوحة
02

Target Selection — اختيار الهدف

يحدد حساباً عالي القيمة: رئيس قسم أو مسؤول شؤون طلاب لديه صلاحية تعديل الدرجات أو الوصول للبيانات المالية.

→ Pattern: hod_[college]_[id] | stdaff_[college]_[id]
03

OSINT Collection — جمع بيانات سؤال الأمان

يبحث في دليل موظفي الجامعة الرسمي أو صفحات LinkedIn و Facebook للحصول على الاسم الكامل وتاريخ الميلاد — كلاهما متاح للعموم.

→ Sources: bsu.edu.eg/staff · LinkedIn · Facebook · Researchgate
04

Password Reset Attack — تنفيذ الهجوم

يفتح صفحة "نسيت كلمة المرور" في mis.bsu.edu.eg، يدخل اسم المستخدم المستهدف، يُجيب على سؤال الأمان باستخدام بيانات OSINT. لا قيود على المحاولات (VULN-003).

→ VULN-002 + VULN-003 exploited simultaneously
05

Account Takeover — الاستيلاء على الحساب

النظام يقبل كلمة المرور الجديدة فوراً. لا تحقق إضافي، لا OTP، لا MFA (VULN-004). المالك الأصلي لا يتلقى أي إشعار بما حدث.

→ VULN-004 enables complete access · VULN-005 allows trivial new password
06

Exploitation & Persistence — الاستغلال المستمر

تسجيل دخول بالهوية المخترقة. الأثر يعتمد على الدور: تعديل الدرجات، تصدير بيانات الطلاب، الوصول للسجلات المالية، انتحال هوية العمداء. لا كشف شذوذ — الوصول مستمر إلى ما لا نهاية.

→ Full compromise · No detection · No time limit
🔑

الخلاصة الجوهرية: هذا الهجوم لا يحتاج إلى أي مهارة برمجية. يحتاج فقط: (1) قناة Telegram عامة + (2) دليل موظفي الجامعة + (3) صفحة "نسيت كلمة المرور". متوسط الوقت الكامل للاختراق: أقل من 15 دقيقة.

06 — تأثير الاستغلال

EXPLOITATION IMPACT

📊 تزوير الدرجات الأكاديمية

تعديل درجات الطلاب عبر حسابات الأساتذة أو رؤساء الأقسام. الأثر على المستوى الجامعي بالكامل وصعوبة بالغة في الاكتشاف أو التراجع لاحقًا.

💰 الوصول للسجلات المالية

الاطلاع على بيانات المصاريف الدراسية لآلاف الطلاب. محاولات محتملة لتعديل حالات الدفع. تعريض بيانات مالية حساسة لمخاطر قانونية.

📁 سرقة البيانات الشخصية (PII)

تصدير قواعد بيانات كاملة تحتوي أسماء الطلاب، أرقامهم القومية، عناوينهم، معلوماتهم الأكاديمية. انتهاك صريح لقوانين حماية البيانات المصرية.

🎭 انتحال الهوية المؤسسية

تسجيل دخول بهوية عميد أو مدير للجامعة. إصدار قرارات أكاديمية مزورة، التلاعب بالجداول الدراسية، التواصل الرسمي بهوية مسؤولين.

📈 تصعيد الصلاحيات

FIX #1: REPLACE KBA WITH OTP

المبدأ الجوهري: أسئلة الأمان تعتمد على "معلومات سرية" — لكنها ليست سرية. OTP يعتمد على وصول مادي لجهاز أو بريد — وهذا سر حقيقي غير قابل للحصول عليه عن بعد.

تدفق Reset الآمن المقترح

1

المستخدم يدخل اسمه أو بريده المسجل

2

توليد OTP عشوائي تشفيريًا

6–8 أرقام، صالح 10 دقائق فقط، مرتبط بالجلسة

3

OTP يُرسَل للبريد/الهاتف المسجل

لا يُعرض على الشاشة أبداً، فقط لقناة آمنة

4

التحقق من OTP ثم السماح بتغيير كلمة المرور

إبطال فوري بعد الاستخدام، إشعار للمالك الحقيقي

secure_otp_reset.pseudocode 
// ✅ منطق Reset الآمن

function initiateReset(username):
  user = db.findUser(username)
  if !user: return genericError()
  // لا تكشف إن كان الحساب موجوداً

  otp = crypto.randomBytes(4).toString('hex')
  ttl = 10 * 60 // 10 دقائق

  db.storeOTP(user.id, hash(otp), ttl)
  sendEmail(user.email, otp) // فقط للبريد

function verifyOTP(username, submittedOTP):
  stored = db.getOTP(username)
  if !stored || stored.expired:
    return INVALID
  if !timingSafeEqual(hash(submittedOTP), stored.hash):
    logFailedAttempt(username)
    return INVALID
  db.invalidateOTP(username) // single use
  return VALID
